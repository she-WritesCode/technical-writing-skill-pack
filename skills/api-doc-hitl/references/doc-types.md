# Document Type Routing

Choose the document type before drafting. Different doc types should not be shaped the same way.

## API overview

Use for:

- explaining what a product or API does
- orienting a new technical reader
- summarizing core concepts before deeper pages

Prioritize:

- what the API does
- who it is for
- how it fits into a workflow
- links to canonical reference pages

## Endpoint or resource reference

Use for:

- documenting a specific endpoint, object, operation, or SDK call

Prioritize:

- exact inputs and outputs
- errors and constraints
- short examples
- accuracy over narrative

## How-to or workflow guide

Use for:

- helping a reader complete a specific task
- teaching a workflow across multiple steps

Prioritize:

- one clear goal
- one consistent example
- only the context required to complete the task

Avoid turning a how-to into a broad conceptual essay.

## Quick reference guide

Use for:

- scan-friendly summaries
- object hierarchies
- field maps
- concise lookup-oriented materials

Prioritize:

- structure
- labeling
- links back to canonical reference pages

## Release note

Use for:

- user-visible changes
- shipped fixes
- changed behavior

Prioritize:

- what changed
- why it matters
- who is affected
- what action is required

## Migration or deprecation note

Use for:

- replacing older APIs, endpoints, or workflows
- communicating timelines and replacements

Prioritize:

- what is changing
- why it is changing
- when the change matters
- how to migrate
- what remains supported

## Feature release or announcement

Use for:

- introducing a new capability with a technical angle
- helping readers understand why a feature exists

Prioritize:

- the user problem
- the new capability
- a concise example
- where to go next for full docs

## Cross-check

Before drafting, ask:

1. Is the reader trying to understand, decide, or do?
2. Does the page need full explanation or just lookup precision?
3. Is this page canonical, or should it defer to another canonical page?
